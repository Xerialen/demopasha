Parser Landscape

Every known QW MVD parser compared against MIMER (pre-north-star) and the demopasha target. Researched 2026-04-12 from the six public GitHub repos. Bottom line: none of them touch 3D geometry — the north-star BSP+loc+demo fusion is uncharted territory for QW tooling.

QW-Group/mvdparser

C · 11 stars · actively maintained (Feb 2025) · 10 contributors
The community standard, maintained alongside MVDSV and ezQuake. Template-driven output to log files + JSON. Extracts per-player end-of-match aggregates: frags, deaths, armor/weapon/powerup pickup counts, per-weapon shots, ping/speed. Uses fragfile.dat for kill attribution.
fragfile kills player stats item pickup counts no hidden messages no KTXstats no BSP no loc no entity tracking no analysis

JoakimSoderberg/mvdparser

C · 2 stars · abandoned (2012) · ancestor of QW-Group fork
The lineage of the QW-Group version, from 2007-2009. Similar stats extraction plus ghost player detection, CTF flag events, and match state machine (countdown/started/overtime). Template-only output. No activity in 13+ years.
fragfile kills player stats CTF flags ghost players no hidden messages no BSP no loc no analysis

vikpe/mvdparser

Rust · crates.io (37k downloads) · actively maintained (Feb 2026)
Clean metadata extraction library. Reads KTXstats v3, player/team info, frags-by-name, serverinfo, prints, match duration. Does not reconstruct game state — svc_playerinfo, packetentities, updatestat are defined in the enum but never consumed. No positions, no entities.
KTXstats v3 serverinfo fragfile kills CTF flags no entity tracking no positions no stat transitions no BSP no analysis

vikpe/mvdparser-js

TypeScript · 53 lines · one-day project (Jun 2024)
Three regex passes over the raw byte stream to extract serverinfo (first 1KB), KTXstats JSON (last 48KB), and match duration. Not a protocol parser — just string scanning. Browser-oriented (WebCrypto for SHA-256).
KTXstats (regex) serverinfo no protocol parsing no BSP no analysis

miltonizer/quakeworld_records

Python · 1 star · abandoned (Dec 2020) · single-day prototype
Correctly implements MVD framing and a subset of svc_* parsing. Tracks positions, weapon drops via STAT_ACTIVEWEAPON, item pickups via bitmask transitions, sound events. But: frag parsing is a TODO stub, entity state is parsed then discarded, no output of any kind. Good protocol documentation in the README.
positions stat transitions entity parse (discarded) frag parsing (TODO) no KTXstats no output no BSP no analysis

vikpe/example-mvdparser-rust

Rust · 53 lines · usage example for vikpe/mvdparser crate
Minimal example app demonstrating the mvdparser crate. Extracts KTXstats v3 JSON and builds a player-name-to-login mapping. Nothing beyond what the library exposes.
KTXstats v3 login mapping example only

vikpe/demolyzer

C# / WPF · 0 stars · code dump (Jul 2025) · author: Cory Vogel (vogelca)
Windows desktop GUI with real-time 2D map playback (pre-rendered BMP map images for ~15 maps). Custom MVD parser tracking positions, full stat transitions, entity respawn timers, and notably detailed backpack economy: RL/LG from enemy vs team packs, donated vs lost. Computes "Power" timeline (health+armor*weapon*powerup per second), TP_Rating composite, and weapon matchup win rates. No export, no CLI — GUI-only with seek/pause/rate controls and time-series charts.
positions stat transitions entity respawn timers backpack economy 2D map playback power timeline BSP (pre-rendered images only) no KTXstats no hidden messages no loc files no export / no CLI Windows only

The Gap

Every parser on this list works from the demo byte stream only. The most advanced (QW-Group) produces end-of-match stat tables. Even pre-north-star, MIMER was already the deepest MVD parser in existence: full state reconstruction, hidden messages, entity tracking, behavioral patterns, resource economics, evidence-backed claims.

The north-star goal of fusing BSP+loc+demo is genuinely uncharted territory for QW tooling. Nobody has built a parser that knows the shape of the rooms, where items sit in 3D space, what's visible from where, or how the geometry constrains movement. That's the leap from "stats extractor" to "understands the game."
